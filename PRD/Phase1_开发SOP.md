## 二、需求开发SOP

### 2.1 PRD评审流程

```
1. PM输出PRD初稿
      ↓
2. 召开PRD评审会议（产品、前端、后端、测试参加）
      ↓
3. 评审内容：
   [ ] 功能范围是否清晰
   [ ] 业务流程是否通顺
   [ ] 状态机是否完整
   [ ] 异常场景是否覆盖
   [ ] 接口需求是否明确
   [ ] 优先级是否合理
      ↓
4. 评审问题记录 → 形成"评审问题清单"
      ↓
5. PM根据问题修订PRD
      ↓
6. 确认评审通过 → 进入设计阶段
```

**评审问题模板**：
| 序号 | 提出人 | 问题 | 答案 | 状态 |
|------|--------|------|------|------|
| 1 | 后端 | 派单逻辑具体怎么实现？ | 距离优先，3km内派单 | 已确认 |

### 2.2 设计评审流程

```
1. UI输出设计稿
      ↓
2. 召开设计评审会议（产品、UI、前端参加）
      ↓
3. 评审内容：
   [ ] 页面结构是否完整
   [ ] 交互是否符合产品需求
   [ ] 组件是否可复用
   [ ] 是否覆盖所有状态
   [ ] 文案是否准确
      ↓
4. 确认通过 → 前端开始页面开发
```

### 2.3 接口评审流程

```
1. 后端输出接口文档初稿
      ↓
2. 召开接口评审会议（前端、后端参加）
      ↓
3. 评审内容：
   [ ] 接口路径是否规范
   [ ] 请求参数是否完整
   [ ] 响应数据结构是否清晰
   [ ] 错误码是否定义
   [ ] 接口是否满足产品需求
      ↓
4. 确认通过 → 前端开始对接
```

## 四、代码管理SOP

### 4.1 分支策略

```
master (正式环境)
  ↑
  ↑
release (预发布环境)
  ↑
  ↑
develop (开发环境)
  ↑
  ↑
feature/xxx (功能分支)
```

### 4.2 Commit规范

```
格式：<type>(<scope>): <subject>

示例：
feat(order): 新增订单创建接口
fix(repair): 修复派单超时问题
docs(README): 更新接口文档
style(UI): 调整订单列表样式
refactor(pay): 重构支付流程
test(order): 新增订单测试用例
```

**Type说明**：
| 类型 | 说明 |
|------|------|
| feat | 新功能 |
| fix | Bug修复 |
| docs | 文档更新 |
| style | 格式调整 |
| refactor | 代码重构 |
| test | 测试相关 |
| chore | 构建/辅助工具 |

### 4.3 Code Review清单

```
Reviewer: xxx
Author: xxx
PR: #xxx

[ ] 代码符合编码规范
[ ] 逻辑正确无明显bug
[ ] 单元测试通过
[ ] 无敏感信息泄露
[ ] 注释清晰
[ ] 命名规范
[ ] 异常处理完整

问题：
- 问题1

结论：✅ 通过 / ❌ 需要修改
```

## 五、接口开发SOP

### 5.1 接口开发流程

```
1. 后端根据PRD设计接口
      ↓
2. 输出接口文档（Swagger/Apifox）
      ↓
3. 接口评审
      ↓
4. 后端实现接口
      ↓
5. 单元测试
      ↓
6. 部署到测试环境
      ↓
7. 前端对接
      ↓
8. 联调测试
      ↓
9. 上线
```

### 5.2 接口文档规范

```yaml
# 示例：创建维修订单接口

## 接口信息
- 路径: /api/orders/repair
- 方法: POST
- 描述: 用户创建维修订单

## 请求参数
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| type | string | 是 | 故障类型ID |
| addressId | string | 是 | 地址ID |
| description | string | 否 | 故障描述 |

## 响应参数
| 参数名 | 类型 | 说明 |
|--------|------|------|
| orderId | string | 订单ID |
| status | string | 订单状态 |
| estimatedPrice | object | 预估价格 |

## 错误码
| 码值 | 说明 |
|------|------|
| 1001 | 参数错误 |
| 1002 | 地址不存在 |
| 1003 | 服务不可用 |
```

## 六、测试SOP

### 6.1 测试用例评审

```
1. 测试输出测试用例初稿
      ↓
2. 召开用例评审（产品、测试、开发参加）
      ↓
3. 评审内容：
   [ ] 用例覆盖率是否完整
   [ ] 用例逻辑是否正确
   [ ] 边界条件是否覆盖
   [ ] 异常场景是否覆盖
      ↓
4. 确认通过 → 执行测试
```

### 6.2 Bug管理

```
Bug等级定义：
- P0: 阻塞性问题，无法继续测试/使用
- P1: 严重问题，功能不可用
- P2: 一般问题，功能可用但有瑕疵
- P3: 轻微问题，不影响使用

Bug处理流程：
1. 测试提交Bug
      ↓
2. 开发确认Bug
      ↓
3. 开发修复Bug
      ↓
4. 测试回归验证
      ↓
5. 关闭Bug
```

### 6.3 上线标准

```
P0 Bug: 0个
P1 Bug: 0个
P2 Bug: ≤3个
P3 Bug: ≤10个

上线前必须满足：
[ ] 所有P0/P1 Bug已修复
[ ] 核心流程测试通过
[ ] 性能指标达标
[ ] 安全扫描通过
```

## 七、上线SOP

### 7.1 上线前检查

```
1. 确认所有测试用例通过
      ↓
2. 确认无P0/P1 Bug
      ↓
3. 确认代码Review通过
      ↓
4. 确认接口文档已更新
      ↓
5. 确认部署脚本就绪
      ↓
6. 确认回滚方案就绪
      ↓
7. 发送上线通知
```

### 7.2 上线流程

```
1. 备份数据库
      ↓
2. 执行数据库迁移
      ↓
3. 部署后端服务
      ↓
4. 部署前端资源
      ↓
5. 验证核心功能
      ↓
6. 监控日志和指标
      ↓
7. 上线完成通知
```

### 7.3 回滚流程

```
触发条件：
- P0 Bug出现
- 核心功能不可用
- 性能严重下降

回滚步骤：
1. 停止部署
      ↓
2. 执行回滚脚本
      ↓
3. 验证回滚成功
      ↓
4. 通知相关人员
      ↓
5. 分析问题原因
      ↓
6. 修复后重新上线
```

## 八、沟通SOP

### 8.1 会议清单

| 会议 | 频率 | 时长 | 参与 |
|------|------|------|------|
| 每日站会 | 每日 | 15min | 全体 |
| 周例会 | 每周 | 30min | PM+技术 |
| 评审会 | 按需 | 60min | 相关人员 |
| 上线复盘 | 上线后 | 45min | 全体 |

### 8.2 文档管理

| 文档 | 位置 | 更新频率 | 负责人 |
|------|------|---------|--------|
| PRD | PRD/ | 需求变更时 | PM |
| 接口文档 | Swagger | 接口开发时 | 后端 |
| 设计稿 | Figma | 设计输出时 | UI |
| 测试用例 | 测试管理工具 | 用例编写时 | 测试 |
| 代码 | Git | 开发时 | 开发 |

### 8.3 紧急响应

```
紧急情况定义：
- 系统崩溃
- 核心功能不可用
- 数据异常
- 安全漏洞

响应流程：
1. 值班人员发现/收到告警
      ↓
2. 5分钟内响应
      ↓
3. 初步判断影响范围
      ↓
4. 紧急拉群（相关人员）
      ↓
5. 制定解决方案
      ↓
6. 执行修复/回滚
      ↓
7. 验证恢复
      ↓
8. 复盘总结
```
