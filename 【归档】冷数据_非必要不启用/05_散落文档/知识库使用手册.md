# 安电通项目知识库使用手册

## 📖 一、知识库是什么？

知识库是一个结构化的项目信息存储系统，包含：
- **17个知识实体**：项目模块、功能、规范等
- **31条知识关系**：模块间的依赖、包含、触发等关系
- **120条观察记录**：每个实体的详细信息

---

## 🔍 二、如何查询知识库

### 方式1：直接查看JSON文件

**最简单的方式**：直接打开 `knowledge_graph.json` 文件

**示例查询**：想知道"订单状态机"有哪些状态？
1. 打开 `knowledge_graph.json`
2. 搜索 "订单状态机"
3. 查看其 `observations` 字段
4. 找到答案：7种状态（Pending、Accepted、Arrived、In Progress、Completed、Paid、Cancelled）

### 方式2：使用Knowledge MCP工具（推荐）

如果MCP配置完成，可以使用以下工具：

#### 1. 搜索知识节点
```
工具：mcp_Knowledge_Graph_Memory_search_nodes
参数：query = "订单状态机"
```

#### 2. 打开特定节点
```
工具：mcp_Knowledge_Graph_Memory_open_nodes
参数：names = ["订单状态机", "支付流程"]
```

#### 3. 读取整个知识图谱
```
工具：mcp_Knowledge_Graph_Memory_read_graph
```

## 💡 三、实际应用场景

### 场景1：快速了解模块功能

**问题**：应急维修模块是做什么的？

**查询方法**：
1. 打开 `knowledge_graph.json`
2. 搜索 "应急维修模块"
3. 查看 `observations` 字段

**答案**：
```
- 核心价值：30分钟极速响应，解决紧急电路故障
- 日均请求量：5000+
- 业务流程：发现故障 → 选择故障类型 → 选择地址 → 提交预约 → 派单 → 电工上门 → 维修 → 支付
- Phase-1优先级：P0（必须）
```

**问题**：派单系统的代码在哪个文件？

**查询方法**：
1. 打开 `knowledge_graph.json`

3. 查看 `observations` 中的 "代码实现"

**答案**：
```
pages/electrician/hall/TaskHall.tsx
```

### 场景3：理解业务流程

**问题**：免费检测的转化逻辑是什么？

**查询方法**：
1. 打开 `knowledge_graph.json`
2. 搜索 "免费检测模块"
3. 查看 `observations` 中的业务流程

**答案**：
```
预约检测 → 电工上门 → 免费检测 → 生成报告 → 发现问题 → 推荐整改 → 付费维修

关键规则：
- 每个用户仅限首次检测免费
- 第二次检测需付费：¥50上门费
- 检测本身不收费，仅整改材料收费
- 首次检测奖励50积分
```

### 场景4：理解模块依赖关系

**问题**：应急维修模块依赖哪些系统？

**查询方法**：
1. 打开 `knowledge_graph.json`
2. 查看 `relations` 部分
3. 找到 `from: "应急维修模块"` 的关系

**答案**：
```
- 应急维修模块 → 派单系统（依赖）
- 应急维修模块 → 订单状态机（依赖）
- 应急维修模块 → 支付流程（依赖）
```

### 场景5：查找文档位置

**问题**：订单状态机的详细文档在哪里？

**查询方法**：
1. 打开 `knowledge_graph.json`
2. 搜索 "订单状态机"
3. 查看 `observations` 中的 "文档位置"

**答案**：
```
项目逻辑分析/03_核心业务/01_订单状态机/订单状态机功能逻辑.md
```

### 场景6：了解异常处理标准

**问题**：P0级异常的响应时间要求？

**查询方法**：
1. 打开 `knowledge_graph.json`
2. 搜索 "异常处理体系"
3. 查看 `observations` 中的响应时间

**答案**：
```
P0（致命异常）：响应时间 <5分钟，立即处理+人工介入
P1（严重异常）：响应时间 <30分钟，自动降级+人工跟进
P2（一般异常）：响应时间 <2小时，自动处理+记录日志
P3（轻微异常）：响应时间 <24小时，记录日志+定期优化
```

## 🛠️ 四、如何维护知识库

### 1. 添加新知识实体

**场景**：项目新增了一个"优惠券系统"模块

**操作步骤**：
1. 打开 `knowledge_graph.json`
2. 在 `entities` 数组中添加新实体：

```json
{
  "name": "优惠券系统",
  "entityType": "功能模块",
  "observations": [
    "对应文件：pages/user/coupons/CouponList.tsx",
    "核心价值：用户优惠、促销活动",
    "优惠券类型：满减券、折扣券、现金券",
    "有效期：领取后30天内有效",
    "Phase-1优先级：P1（扩展）",
    "文档位置：项目逻辑分析/01_用户端/模块8_个人中心/02_优惠券系统/"
  ]
}
```

### 2. 更新现有实体

**场景**：订单状态机新增了"超时自动取消"功能

**操作步骤**：
1. 打开 `knowledge_graph.json`
2. 找到 "订单状态机" 实体
3. 在 `observations` 数组中添加新观察：

```json
"observations": [
  "7种订单状态：Pending、Accepted、Arrived、In Progress、Completed、Paid、Cancelled",
  "状态流转：用户下单 → 待接单 → 已接单 → 已到达 → 施工中 → 已完成 → 已支付",
  "异常路径：超时/取消 → Cancelled",
  "代码实现：context/AppContext.tsx中的updateOrder方法",
  "状态变更时发送通知给用户",
  "Phase-1优先级：P0（核心）",
  "文档位置：项目逻辑分析/03_核心业务/01_订单状态机/订单状态机功能逻辑.md",
  "新增功能：订单超时自动取消（2024-02-01）"
]
```

### 3. 建立新关系

**场景**：优惠券系统与支付流程建立了关联

**操作步骤**：
1. 打开 `knowledge_graph.json`
2. 在 `relations` 数组中添加新关系：

```json
{
  "from": "优惠券系统",
  "to": "支付流程",
  "relationType": "关联"
}
```

## 📊 五、知识库结构说明

### 实体类型分类

| 实体类型 | 说明 | 示例 |
|---------|------|------|
| 项目 | 项目总览 | 安电通项目 |
| 核心模块 | 核心业务模块 | 核心业务模块 |
| 子模块 | 核心模块的子模块 | 订单状态机、支付流程 |
| 端模块 | 用户端/电工端 | 用户端模块、电工端模块 |
| 功能模块 | 具体功能模块 | 地图模块、商城模块 |
| 业务模块 | 业务流程模块 | 应急维修模块、免费检测模块 |
| 管理系统 | 后台管理系统 | 运营后台模块 |
| 技术规范 | 技术标准规范 | API接口规范、异常处理体系 |

### 关系类型说明

| 关系类型 | 说明 | 示例 |
|---------|------|------|
| 包含 | 模块层级关系 | 安电通项目 → 核心业务模块 |
| 依赖 | 模块间依赖关系 | 应急维修模块 → 派单系统 |
| 触发 | 业务触发关系 | 支付流程 → 积分体系 |
| 更新 | 状态更新关系 | 派单系统 → 订单状态机 |
| 管理 | 后台管理关系 | 运营后台模块 → 订单状态机 |
| 定义 | 规范定义关系 | 安电通项目 → API接口规范 |
| 约束 | API约束关系 | API接口规范 → 核心业务模块 |
| 保障 | 异常处理保障关系 | 异常处理体系 → 核心业务模块 |
| 关联 | 一般关联关系 | 商城模块 → 积分商城模块 |

## 📝 八、常见问题

### Q1：知识库信息过时了怎么办？
**A**：直接编辑 `knowledge_graph.json` 文件，更新对应的 `observations` 字段

### Q2：如何知道某个模块依赖哪些其他模块？
**A**：查看 `relations` 部分，找到 `from` 为该模块的所有关系

### Q3：如何快速找到某个功能的代码位置？
**A**：搜索对应实体，查看 `observations` 中的 "代码实现" 或 "对应文件"

### Q4：知识库可以导出吗？
**A**：可以！`knowledge_graph.json` 就是标准JSON格式，可以导入到任何支持JSON的工具中

### Q5：如何添加新的实体类型？
**A**：直接在 `entityType` 字段中使用新的类型名称即可
