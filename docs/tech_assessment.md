## 二、技术债务清单

### 2.1 高优先级技术债务 (P0)

#### TD-001: Context架构重构

**现状**: AppContext.tsx 承载用户、订单、地址、消息、积分等所有状态

**问题**:
- 单一Provider导致不必要的重渲染
- 状态逻辑混杂，难以维护
- 缺乏持久化策略，sessionStorage数据易丢失

**建议方案**:
```typescript
// 拆分为多个Context
UserContext      // 用户信息、认证状态
OrderContext     // 订单列表、当前订单
AddressContext   // 地址管理
MessageContext   // 消息通知
PointsContext    // 积分体系

// 使用 Zustand 或 Jotai 替代 Context
// 提供更好的性能和持久化支持
```

**工作量**: 2-3人周

---

#### TD-002: API服务层建设

**现状**: 无统一API请求层，组件直接调用fetch，Mock数据硬编码

**问题**:
- 缺乏请求拦截/响应拦截
- 无统一错误处理
- 无请求取消机制
- Mock数据与生产代码混杂

**建议方案**:
```typescript
// services/api/
├── request.ts        // 基于axios的请求封装
├── interceptors.ts   // 拦截器(认证/日志/错误)
├── endpoints.ts      // API端点定义
├── mock/             // Mock数据层(开发环境)
│   ├── mockUser.ts
│   ├── mockOrder.ts
└── types/            // API响应类型定义
```

**工作量**: 1-2人周

---

### 2.2 中优先级技术债务 (P1)

#### TD-003: 组件拆分与复用

**现状**: 大型组件(UserHome.tsx 300行、AIChatOverlay.tsx 500行)，共享组件仅4个

**问题**:
- 组件复杂度高，难以维护
- 重复代码模式(卡片、列表、表单)
- 缺乏UI组件库

**建议方案**:
```
components/
├── ui/               # 基础UI组件
│   ├── Button.tsx
│   ├── Card.tsx
│   ├── Input.tsx
│   ├── Modal.tsx
│   └── Loading.tsx
├── business/         # 业务组件
│   ├── OrderCard.tsx
│   ├── ElectricianCard.tsx
│   ├── ServiceGrid.tsx
│   └── AddressSelector.tsx
└── layout/           # 布局组件
    ├── PageLayout.tsx
    ├── TabLayout.tsx
```

**工作量**: 3-4人周

---

#### TD-004: 测试体系建设

**现状**: 无单元测试、无E2E测试、无CI/CD测试流程

**问题**:
- 代码变更无保障
- Bug难以提前发现
- 回归成本高

**建议方案**:
```yaml
测试体系:
  单元测试: Vitest + @testing-library/react
  E2E测试: Playwright
  CI集成: GitHub Actions / Jenkins
  覆盖目标: 核心业务逻辑 > 80%
```

**工作量**: 2人周(搭建) + 持续投入
